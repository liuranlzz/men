# 剪映草稿自动生成调研与技术方案

调研日期：2026-06-16

## 目标

本项目的目标不是直接生成完整成片，而是自动生成一个可以在剪映中打开、继续人工编辑的草稿：

- 从项目中的口稿 Markdown 生成剪映草稿，例如 `content/遥控4.0/遥控4.0内容.md`
- 草稿中包含 AI 配音音轨
- 草稿中包含字幕轨道
- 字幕字号、颜色、描边、阴影、位置、自动换行等样式可控
- 视频画面可以先用占位图、白底、截图或后续素材补齐，不作为第一阶段硬要求

## 结论

推荐方案：本地 Python 草稿生成器 + 外部 TTS + `pyJianYingDraft`。

核心链路：

```text
口稿 Markdown
  -> 解析分段、时间轴、口播文本
  -> TTS 生成 mp3/wav 配音
  -> 生成 SRT 字幕或直接生成 TextSegment
  -> pyJianYingDraft 写入剪映草稿
  -> 剪映中打开草稿，人工补画面/微调/导出
```

不推荐把剪映官方 API 作为主方案。目前公开资料中没有看到剪映专业版/CapCut 提供稳定的“创建本地剪映草稿”的官方开发者 API，也没有查到官方剪映 MCP。剪映/CapCut 有自动字幕、文本转语音、字幕样式编辑等产品能力，但主要是 UI 功能，不是面向代码调用的草稿生成接口。

Coze 可以作为上游编排层，但不应该作为本地草稿写入层。合理方式是让 Coze 生成结构化 JSON、分镜、字幕、TTS 音频 URL，然后由本机脚本下载资源并写入剪映草稿目录。

## 调研发现

### 剪映 / CapCut 官方能力

CapCut 官方页面提供了文本转语音能力，页面说明支持 200+ AI voices、语速、音量、pitch 等参数调节，也支持下载音频或应用到项目中。参考：<https://www.capcut.com/tools/text-to-speech>

CapCut 官方字幕页面提供自动字幕、字幕模板和字幕样式编辑能力，页面说明可以自动识别音频生成字幕，字幕模板可自定义字体、颜色、大小、动画等。参考：

- <https://www.capcut.com/tools/ai-caption-generator>
- <https://www.capcut.com/tools/auto-caption-generator>

这些能力证明“剪映生态本身支持 AI 配音和可编辑字幕”，但没有证明它们能通过官方 API 写入本地剪映草稿。第一阶段不要依赖“调用剪映内置 TTS/字幕 API”，否则会卡在未公开接口、登录态、风控、UI 变化和版本兼容上。

### 开源草稿生成方案

最值得采用的项目是 `GuanYixuan/pyJianYingDraft`：

- 仓库：<https://github.com/GuanYixuan/pyJianYingDraft>
- 定位：Python 剪映草稿生成及导出工具
- 支持：视频、图片、音频、文本、字幕、轨道、转场、滤镜、特效、关键帧等
- 字幕：支持导入 `.srt`，并批量设置文本样式
- 文本样式：支持字号、颜色、字体、描边、背景、阴影、自动换行
- 位置：通过文本片段的 `ClipSettings` 控制，例如 `transform_y`
- 安装：README 中给出的方式是 `pip install pyJianYingDraft`

重要限制：

- 模板模式依赖未加密的 `draft_content.json`。项目 README 提到剪映 6+ 对草稿文件进行了加密，所以“加载已有模板草稿并替换内容”目前主要支持剪映 5.9 及以下。
- 批量导出依赖 Windows UI automation，且剪映 7+ 隐藏了部分控件。我们第一阶段只生成草稿，不自动导出，所以这个限制可以先绕开。
- 生成“新草稿”的能力比“读取高版本模板草稿”更稳。建议第一阶段不要做复杂模板复刻。

其他相关项目：

- `notinmood/JianyingDraft.PY`：根据资源文件自动生成剪映草稿。<https://github.com/notinmood/JianyingDraft.PY>
- `xiaoyiv/JianYingProDraft`：快速创建剪映草稿。<https://github.com/xiaoyiv/JianYingProDraft>
- `ShaunZh/jianYingDraftCreate`：根据 Coze 工作流生成的数据生成剪映草稿。<https://github.com/ShaunZh/jianYingDraftCreate>

### Coze 工作流可行性

Coze 适合做这些事情：

- 根据主题或现有口稿生成结构化分镜
- 把 Markdown 口稿转换成 JSON
- 调用模型改写口播文案
- 生成图片提示词
- 调用 TTS/图片/素材相关插件，输出资源 URL
- 输出统一格式：`image_list`、`audio_list`、`text_cap`、`text_timelines`

但 Coze 不适合直接做这些事情：

- 写入本机 `~/Movies/JianyingPro/...` 草稿目录
- 直接控制本地剪映专业版
- 依赖剪映本地草稿格式做最终落盘

社区项目 `ShaunZh/jianYingDraftCreate` 的模式很接近可落地方案：从 Coze 工作流复制 JSON，脚本下载/缓存图片和音频，然后创建剪映草稿并导入字幕样式。这个模式说明 Coze 可以负责“云端内容生产”，本地脚本负责“剪映草稿落地”。

## 技术方案对比

| 方案 | 描述 | 优点 | 风险 | 结论 |
|---|---|---|---|---|
| A. 本地脚本 + pyJianYingDraft + 外部 TTS | 本地解析口稿、生成配音、生成字幕、写剪映草稿 | 可控、可迭代、无剪映官方 API 依赖、适合当前目标 | 需要校准字幕样式；需选择 TTS 服务 | 推荐主方案 |
| B. Coze 工作流 + 本地草稿适配器 | Coze 输出 JSON 和资源 URL，本地脚本写草稿 | 适合后期内容批量化和工作流编排 | 仍需要本地脚本；调试链路更长 | 第二阶段接入 |
| C. 自动化剪映 UI | 用脚本点击剪映生成字幕/TTS | 可复用剪映内置能力 | UI 易变、登录/会员/风控、稳定性差 | 不推荐主方案 |
| D. 高版本剪映模板替换 | 复制已有精美模板，替换文本/音频 | 成片风格一致 | 剪映 6+ 草稿加密，读取模板风险高 | 暂缓 |
| E. 直接生成 MP4 | 用 ffmpeg/moviepy 合成视频 | 完全可控，不依赖剪映 | 不是可编辑剪映草稿 | 不是当前目标 |

## 推荐架构

### 1. 口稿解析层

输入文件示例：`content/遥控4.0/遥控4.0内容.md`

当前口稿格式已经比较规整：

```text
#### 【0-6秒】 黄金开头
- **画面**：...
- **知性女声**：
    “...”
```

解析后建议生成统一中间结构：

```json
{
  "project_name": "遥控4.0",
  "canvas": { "width": 1080, "height": 1920 },
  "segments": [
    {
      "start": 0.0,
      "end": 6.0,
      "scene_title": "黄金开头：痛点展示",
      "visual_note": "实拍手部特写...",
      "voice_role": "知性女声",
      "voice_text": "新配的卷帘门遥控器，按键功能全乱了？千万别扔！今天教你不用改线路、不拆电机，一招轻松调回来！",
      "caption_text": "新配的卷帘门遥控器，按键功能全乱了？千万别扔！今天教你不用改线路、不拆电机，一招轻松调回来！"
    }
  ]
}
```

第一版可以按口稿中写死的时间段生成字幕和音频位置。后续如果 TTS 实际时长和口稿时间段差异很大，再做自动重排：

- 固定原口稿时间轴：适合快速生成可编辑草稿
- 按 TTS 实际时长重排：适合追求音画精准

### 2. TTS 层

不要调用剪映内置 TTS。第一阶段直接选一个外部 TTS 服务，生成本地音频文件，再把音频导入剪映草稿。

候选：

- Edge TTS：适合 POC，成本低，上手快，但属于非正式生产依赖
- 火山引擎 / 字节系 TTS：中文声音质量和国内可用性较好，适合生产化
- MiniMax / ElevenLabs / OpenAI TTS：声音表现好，适合多风格测试，但需要确认费用、授权、国内网络和 API 稳定性

第一阶段建议：

- POC 默认使用 Edge TTS 或已有可用 TTS Key
- 每个分段生成一个音频文件：`audio_000.mp3`、`audio_001.mp3`
- 文件名、文本、voice、duration 写入 manifest，便于复用缓存
- 如果有口稿时间轴，则音频片段按 `start` 放入音轨

### 3. 字幕层

有两种生成方式：

1. 生成 SRT，再用 `script.import_srt(...)` 导入并批量套样式
2. 直接创建多个 `TextSegment`，每条字幕独立控制样式和位置

第一阶段推荐 SRT 方式，理由是结构简单、易调试、容易复用。

字幕样式通过 `pyJianYingDraft` 控制：

```python
style_ref = TextSegment(
    "template",
    trange(0, 1_000_000),
    style=TextStyle(
        size=8.0,
        align=1,
        auto_wrapping=True,
        color=(1.0, 1.0, 1.0),
    ),
    border=TextBorder(
        alpha=1.0,
        color=(0.0, 0.0, 0.0),
        width=45.0,
    ),
)

script.import_srt(
    "captions.srt",
    track_name="subtitles",
    style_reference=style_ref,
    clip_settings=ClipSettings(transform_y=-0.78),
)
```

需要重点校准的参数：

- `TextStyle.size`：剪映内部字号不是 CSS px，需要用实际草稿视觉校准
- `ClipSettings.transform_y`：通常是归一化坐标，底部字幕可从 `-0.75 ~ -0.85` 试起
- `auto_wrapping` 和 `max_line_width`：避免长句超出屏幕
- 描边宽度、阴影：短视频字幕可读性关键
- 字体：如果要求固定字体，需要确认剪映和系统都能识别

### 4. 草稿生成层

草稿生成建议使用 `pyJianYingDraft.ScriptFile(1080, 1920)` 创建竖屏 9:16 草稿：

- 视频轨：先放白底占位图或每段画面提示卡
- 音频轨：放每段 TTS mp3
- 字幕轨：导入 SRT 并设置统一样式
- 元信息：生成 `draft_content.json`、`draft_info.json`、必要的 layout/config 文件
- 输出位置：剪映草稿目录

macOS 常见路径：

```text
~/Movies/JianyingPro/User Data/Projects/com.lveditor.draft
```

如果用户本机路径不同，应在配置文件中显式设置：

```yaml
jianying_draft_root: "/Users/<user>/Movies/JianyingPro/User Data/Projects/com.lveditor.draft"
canvas:
  width: 1080
  height: 1920
subtitle:
  size: 8.0
  transform_y: -0.78
  color: [1.0, 1.0, 1.0]
  border_color: [0.0, 0.0, 0.0]
  border_width: 45.0
tts:
  provider: edge
  voice: zh-CN-XiaoxiaoNeural
```

## 版本策略

第一阶段只生成新草稿，不读取高版本剪映模板。

这样可以避开最麻烦的加密问题。模板复刻虽然诱人，但会碰到剪映 6+ 的 `draft_content.json` 加密限制。等 POC 跑通后，如果确实需要固定片头、贴纸、花字、合集箭头等复杂效果，再做两条路：

- 用代码直接生成常用素材和文本效果
- 准备低版本剪映模板，走 `pyJianYingDraft` 的模板模式

## 建议落地步骤

### POC 1：最小可打开草稿

输入：`content/遥控4.0/遥控4.0内容.md`

输出：

- 一个剪映草稿目录
- 一条白底或占位图视频轨
- 一条 AI 配音音轨
- 一条字幕轨
- 字幕位置固定在底部
- 字幕字号、描边、颜色可通过配置调整

验收：

- 剪映列表能看到草稿
- 草稿能打开
- 音频能播放
- 字幕出现且时间大致正确
- 调配置后重新生成，字幕样式能变化

### POC 2：字幕样式校准

固定一份 5-10 秒测试口稿，生成多组样式：

- 字号：`6.0`、`8.0`、`10.0`
- 位置：`transform_y=-0.70/-0.78/-0.85`
- 描边：`30/45/60`
- 最大行宽：`0.75/0.85/0.95`

在剪映中选定一套“项目默认短视频字幕样式”，写入配置。

### POC 3：Coze 接口预留

本地脚本先接受统一 JSON 输入：

```bash
python tools/jianying_draft/create_draft.py \
  --input content/遥控4.0/遥控4.0内容.md \
  --config config/jianying_draft.yaml
```

后续再增加：

```bash
python tools/jianying_draft/create_draft.py \
  --input coze_output.json \
  --input-format coze
```

这样 Coze 是否接入不会影响本地核心能力。

```text
tools/

    create_draft.py
    parse_script_md.py

    subtitle.py
```

第一版不建议直接把生成的草稿目录放进 Git。草稿里会包含大量素材和机器路径，应作为本地生成物。

## 主要风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| 剪映没有官方草稿 API | 无法走官方稳定接口 | 使用开源草稿写入方案，保留版本测试 |
| 剪映草稿格式变化 | 生成草稿可能打不开 | 锁定本机剪映版本，建立 golden draft 回归样例 |
| 高版本模板加密 | 不能读取复杂模板 | 第一阶段只生成新草稿；复杂样式后置 |
| 字幕字号与位置不是像素语义 | 视觉需要校准 | 做字幕样式矩阵测试，固化配置 |
| TTS 音频时长和口稿时间段不一致 | 字幕和配音可能错位 | POC 先用口稿时间轴；后续按实际音频重排 |
| TTS 服务授权/费用/网络不确定 | 生产不稳定 | TTS 做 provider 插件化，保留缓存 |
| Coze 无法写本地文件 | 不能直接生成草稿 | Coze 只产 JSON，本地脚本落盘 |
| 自动导出受剪映版本和系统限制 | 批量出片可能不稳 | 当前不做自动导出，保留人工导出 |

## 技术选择

当前建议选择：

- 草稿生成：`pyJianYingDraft`
- 口稿解析：本地 Python 正则/Markdown 解析
- TTS：先用低成本方案 POC，生产前再换正式 TTS
- 字幕：先生成 SRT，再用 `import_srt` 批量套样式
- Coze：暂不作为第一阶段必需依赖，只预留 JSON 输入格式
- 导出：第一阶段人工在剪映中导出

这个选择的好处是最贴近目标，并且每一层都可替换：

- TTS 可以换
- Coze 可以加
- 字幕样式可以调
- 视频素材可以从占位图升级到图片/实拍素材
- 后续可以再增加自动导出
